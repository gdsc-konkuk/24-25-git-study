# Git Study Notes (Study 02)

## 1. Revision Selection (7.1)
- Git에서 리비전을 선택하는 다양한 방법을 제공합니다. `HEAD`, `HEAD~1`, `HEAD@{yesterday}`와 같은 형식을 사용하여 특정 커밋을 선택할 수 있습니다.
- 이를 통해 시간 기반 또는 부모 관계에 따라 특정 커밋을 쉽게 참조할 수 있습니다.

## 2. Interactive Staging (7.2)
- `git add -i` 옵션을 사용하여 단계적으로 변경 사항을 추가할 수 있습니다.
- Hunk 단위로 선택하여 커밋하거나 특정 변경 사항만 선택할 수 있어, 커밋을 세밀하게 관리할 수 있습니다.

## 3. Stash (7.3)
   - **Push**: 현재 작업 중인 변경 사항을 임시로 저장하는 기능입니다. 작업 중이던 내용을 안전하게 보관하고, 이후 다시 이어서 작업할 수 있습니다.
   - **Apply & Pop**: 저장된 내용을 작업 디렉터리에 복원합니다. `apply`는 내용 복원 후 그대로 두고, `pop`은 복원 후 스택에서 제거합니다.
   - **Drop**: 필요 없는 stash 항목을 삭제하여 저장 공간을 정리합니다.

## 4. Bisect (7.5)
- `git bisect`는 문제를 찾기 위한 이진 검색 도구입니다. 문제가 발생한 커밋을 빠르게 찾을 수 있도록, 특정 범위 내에서 커밋을 반으로 나누어 테스트하며 문제의 원인을 추적합니다.

## 5. Rebase Deep Dive (7.6)
   - **Commit 합치기**: `git rebase -i` 명령을 통해 여러 커밋을 하나로 합칠 수 있습니다. 커밋 히스토리를 깔끔하게 정리하는 데 유용합니다.
   - **Commit 분리하기**: 기존 커밋을 쪼개어 두 개의 커밋으로 분리할 수 있습니다. 이를 통해 커밋 내용을 더 명확하게 나눌 수 있습니다.
   - **`-onto` 옵션**: 리베이스 시 특정 브랜치나 커밋을 기준으로 변경 사항을 적용할 수 있는 옵션입니다. 이를 통해 브랜치를 재구성할 수 있습니다.

## 6. Reset & Checkout 작동 방식 (7.7)
- **Reset**: 특정 커밋으로 되돌아가고, 그 이후의 커밋을 삭제하거나 임시로 유지할 수 있습니다. `--soft`, `--mixed`, `--hard` 옵션을 통해 스테이징과 작업 디렉터리에 영향을 줄 수 있습니다.
- **Checkout**: 브랜치나 커밋을 선택하여 그 시점의 상태로 작업 디렉터리를 변경합니다. 특정 브랜치를 전환하거나 과거 커밋을 임시로 확인할 때 사용됩니다.

## 1. Revision Selection (7.1) – 특정 커밋 선택

Revision Selection은 특정 시점의 커밋을 선택해서 돌아가거나 확인할 수 있는 기능입니다. 예를 들어, `add-tina1lure-directory` 브랜치에서 작업을 하고 있을 때, 이전 커밋으로 돌아가서 상태를 확인하고 싶을 때 사용합니다.

### 예시
- `HEAD`: 현재 위치한 커밋을 가리킵니다.
- `HEAD~1`: 바로 이전 커밋을 뜻합니다.
- `HEAD@{yesterday}`: 어제 시점의 커밋을 볼 수 있습니다.

```bash
git show HEAD          # 현재 커밋 보기
git show HEAD~1        # 한 단계 이전 커밋 보기
git show HEAD@{yesterday} # 어제의 커밋 보기
```

### 예시 상황
`add-tina1lure-directory` 브랜치에서 `tina1lure` 폴더를 생성하고 커밋했는데, 이전에 커밋한 내용으로 돌아가서 확인하고 싶을 때 `HEAD~1`을 사용하여 바로 이전 상태를 확인할 수 있습니다. 이렇게 특정 시점으로 돌아가면서 변경 사항을 검토할 수 있습니다.

## 2. Interactive Staging (7.2) – 선택적으로 변경 사항 추가하기

Interactive Staging은 커밋에 추가할 파일이나 특정 변경 사항을 세밀하게 선택할 수 있는 기능입니다. `tina1lure` 폴더 안에 여러 파일을 추가하고, 그 중 일부만 커밋하고 싶을 때 유용합니다.

### 예시
- `git add -i`: 인터랙티브 모드로 변경 사항을 하나씩 선택해서 추가할 수 있습니다.
- `git add -p`: 파일 내 특정 부분만 선택하여 추가할 수 있습니다.

```bash
git add -i  # 단계적으로 선택해서 스테이징
git add -p  # 변경 사항 중 일부만 선택해서 추가
```

### 예시 상황
`week2.md` 파일을 포함한 여러 파일을 `tina1lure` 폴더에 만들었고, 그 중 일부만 커밋하고 싶다고 가정해 봅시다. `git add -p` 명령어로 파일을 한 덩어리씩 추가하여, 특정 부분만 선택적으로 커밋할 수 있습니다.

## 3. Stash (7.3) – 작업 임시 저장하기

Stash는 작업 중인 변경 사항을 임시로 저장하는 기능입니다. 예를 들어, `add-tina1lure-directory` 브랜치에서 작업하다가 다른 브랜치로 이동해야 할 때, 작업 내용을 임시로 저장하고 나중에 다시 불러와 이어서 작업할 수 있습니다.

### 예시
- `git stash push`: 현재 작업 내용을 임시로 저장합니다.
- `git stash apply`: 임시 저장한 변경 사항을 다시 가져옵니다.
- `git stash pop`: 저장된 작업을 가져오고 스택에서 삭제합니다.
- `git stash drop`: 필요 없는 Stash 항목을 삭제합니다.

```bash
git stash push -m "tina1lure 작업 임시 저장"  # 변경 사항 임시 저장
git stash apply                              # 임시 저장한 변경 사항 다시 가져오기
git stash pop                                # 임시 저장한 변경 사항 가져오고 삭제
git stash drop                               # 필요 없는 임시 저장 삭제
```

### 예시 상황
`add-tina1lure-directory` 브랜치에서 `tina1lure` 폴더에 파일을 작성하다가, 갑자기 `main` 브랜치로 돌아가서 다른 작업을 해야 할 상황이 생겼습니다. `git stash`로 현재 작업 중인 내용을 임시로 저장한 후, `main` 브랜치로 이동하여 작업을 마친 다음 `git stash apply`로 임시 저장한 내용을 불러와서 이어서 작업할 수 있습니다.

## 4. Bisect (7.5) – 문제 발생 커밋 찾기

프로젝트를 하다 보면 특정 커밋에서 문제가 발생할 때가 있습니다. 이런 상황에서 `git bisect` 명령을 사용해 어떤 커밋에서 문제가 발생했는지 빠르게 찾을 수 있습니다.

### 예시
- `git bisect start`: bisect 모드를 시작합니다.
- `git bisect bad`: 현재 커밋이 문제를 포함한다고 표시합니다.
- `git bisect good <커밋 해시>`: 정상적으로 작동하던 커밋을 지정합니다.

```bash
git bisect start               # bisect 시작
git bisect bad                 # 현재 커밋이 문제를 포함함
git bisect good <커밋 해시>    # 정상 작동했던 커밋 지정
git bisect reset               # bisect 종료
```

### 예시 상황
예를 들어, `add-tina1lure-directory` 브랜치에서 여러 커밋을 만들었는데, 어느 순간 `tina1lure` 폴더에서 오류가 발생했다고 가정합니다. 이때 `git bisect`를 사용해 정상 작동했던 커밋과 문제가 생긴 커밋을 지정하여, 문제의 원인이 된 커밋을 빠르게 찾을 수 있습니다.

## 5. Rebase Deep Dive (7.6) – 커밋 정리 및 재구성

Rebase는 여러 커밋을 하나로 합치거나 커밋 히스토리를 정리하는 데 유용합니다. 예를 들어, `add-tina1lure-directory` 브랜치에서 작업 중 여러 번 커밋했지만, 이 커밋들을 하나로 합치고 싶을 때 사용합니다.

### 예시
- `git rebase -i HEAD~3`: 최근 3개의 커밋에 대해 인터랙티브 리베이스를 실행합니다.
- `-onto` 옵션: 특정 커밋이나 브랜치를 기준으로 리베이스할 때 사용합니다.

```bash
git rebase -i HEAD~3           # 최근 3개의 커밋을 합치거나 정리
git rebase --onto <브랜치명> <시작 커밋> <대상 브랜치>
```

### 예시 상황
`add-tina1lure-directory` 브랜치에서 여러 번 커밋을 했다고 가정해봅시다. 나중에 이 커밋들을 깔끔하게 하나의 커밋으로 합치고 싶을 때, `git rebase -i HEAD~3`를 사용해 마지막 3개의 커밋을 선택하여 `pick`을 `squash`로 변경하여 커밋을 하나로 합칠 수 있습니다. 이러면 히스토리가 깔끔해집니다.

## 6. Reset & Checkout (7.7) – 커밋 되돌리기와 브랜치 전환

Reset은 특정 커밋으로 돌아가고, 이후 변경 사항을 삭제하거나 보관하는 기능입니다. Checkout은 특정 브랜치나 커밋으로 이동하여 그 시점의 상태를 확인할 때 사용됩니다.

### 예시
- `git reset`: 특정 커밋으로 돌아갑니다. `--soft`, `--hard` 옵션으로 커밋 상태를 조정할 수 있습니다.
- `git checkout`: 특정 커밋이나 브랜치로 이동하여 상태를 확인합니다.

```bash
git reset --soft HEAD~1  # 마지막 커밋을 되돌리지만, 파일은 그대로 유지
git reset --hard HEAD~1  # 마지막 커밋을 되돌리고, 파일도 삭제

git checkout <커밋 해시>  # 특정 커밋으로 이동하여 상태 확인
git checkout main         # 다시 main 브랜치로 돌아오기
```

### 예시 상황
`add-tina1lure-directory` 브랜치에서 마지막 커밋을 되돌려야 할 때 `git reset --soft HEAD~1` 명령어를 사용하면 커밋이 취소되지만, 파일은 그대로 유지됩니다. 이후 변경 내용을 수정한 후 다시 커밋할 수 있습니다. `git checkout main`을 사용하면 `main` 브랜치로 돌아가 이전 작업 상태를 확인할 수도 있습니다.
